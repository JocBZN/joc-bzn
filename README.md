# JOC-BZN-Mobile 🎮

A mobile **survivors-like** (bullet-heaven) game — think *Vampire Survivors* / *Brotato* with a **cyberpunk** theme. Built in **Godot 4.7** with GDScript, targeting **Android**.

## Game concept
- Top-down arena. The **player** appears centered (a `Camera2D` follows them) but moves freely across a large world.
- **Enemies** spawn continuously from off-screen and chase the player.
- The player **auto-fires** projectiles at the nearest enemy. Enemies have HP and die.
- Enemies touching the player deal **contact damage**; the player has HP + a health bar.
- **Roguelike core (planned):** dead enemies drop **XP** → XP bar fills → **level up** → choose **1 of 3 items/upgrades**. Difficulty ramps over time (enemies faster/tougher, drop more XP).

## Tech & conventions
- **Engine:** Godot 4.7 stable · 2D · "Mobile" renderer. Stretch mode `canvas_items`, aspect `expand` (scales to any phone screen).
- **Language:** GDScript.
- **Indentation: TABS.** Godot rejects mixed tabs/spaces — this is *the* most common error when pasting code. After any paste use **Edit → Convert Indent to Tabs**.
- **Groups** are used to find nodes across the tree: the player is in group `"player"`, enemies in group `"enemy"`. Look up with `get_tree().get_first_node_in_group("player")` and `get_nodes_in_group("enemy")`.
- `get_*_in_group` returns a generic `Node`, so **cast with `as Node2D`** before touching `global_position` (otherwise "cannot infer type" errors). Dynamic property access on such nodes yields harmless **yellow** warnings (yellow = warning/OK, red = error/must fix).

## Project structure
All scenes (`.tscn`) and scripts (`.gd`) live in the project root.

- **`main.tscn`** — the game world (root `Node2D` "Main"):
  - `Ground` (Sprite2D + `ground.gd`) — the infinite repeating grass that follows the player.
  - `World` (Node2D, `y_sort_enabled`) — depth-sorted container holding `Props` (trees), `Rocks` (stones), `Player`, and (added at runtime) the enemies, so they overlap correctly by depth.
  - `Props` (`props.gd`) / `Rocks` (`rocks.gd`) / `DesertStructures` (`desert_structures.gd`) — chunk-based procedural, deterministic spawners (infinite). Rectangle hitboxes with per-side tuning (`hitbox_north/south/east/west`), `sort_anchor`, `min_gap_hitboxes` (min spacing). **Biome rules:** trees avoid the desert *and* its soft gradient; rocks avoid only the hard desert (they may sit on the gradient); desert structures (cactus/house/monument) spawn *only* in the desert — see `BiomeMap.is_desert_chunk` / `desertness_at_chunk` / `desert_inset_chunk`.
  - `Spawner` (Node + `spawner.gd`) — a Timer that instances enemies **in front of** the player (into `World`): since 2026-07-22 they only appear inside a cone of ±`spawn_cone_deg` (**45°**) around the direction the player is facing, at `spawn_distance` (700px).
  - `HUD` (CanvasLayer + `hud.gd`) — screen-fixed UI: health bar + XP bar + level, all built in code.
  - `LevelUp` (CanvasLayer + `levelup.gd`) — the level-up choice screen (3 of **47** upgrades, **weighted by rarity**), styled like *Megabonk*: an ornate `Menu.png` panel with each choice framed by a **rarity border** (Common→Legendary) + matching colored text; pauses the game.
    - ⚠️ **Rarity is a real drop rate, not just a colour** — `RARITY_CHANCE` at the top of `levelup.gd`: Common 30 · Uncommon 30 · Rare 20 · Epic 15 · Legendary 5. The roll picks the **rarity first**, then an item within it, so **how many items a tier holds no longer changes how often that tier appears** — adding a Legendary keeps Legendary at 5% and simply splits it further. (Before this, selection was uniform over the whole list, which made Legendary show up in 11.4% of rows.) Each entry is one dict in `UPGRADES` (id / name / icon / rarity / description) and its real effect is a `match` arm in `_apply()` — **the description text is not the source of truth, `_apply()` is.**
- **`player.tscn`** (`CharacterBody2D` + `player.gd`) — has an **AnimatedSprite2D** (8-directional run + idle poses, `player_frames.tres`), CollisionShape2D, Camera2D. Handles arrow-key movement, auto-fire at nearest enemy (Timer), HP + a contact-damage tick, the fire trail (Firewalker), and death (opens the Game Over screen).
- **`enemy.tscn`** (`CharacterBody2D` + `enemy.gd`) — chases the player, has HP, `take_damage()`, dies via `queue_free()`. Uses an **AnimatedSprite2D** with **8-directional walk animations** (`enemy_frames.tres`, 6 frames each, picked by angle in `enemy.gd`'s `DIRECTII`); the art lives in `homeless directii/frames/`.
- **`bullet.tscn`** (`Area2D` + `bullet.gd`) — flies in a direction, on `body_entered` damages bodies in group `"enemy"`, self-destructs after `lifetime`. Supports **pierce**, **knockback**, **crit**, and **explosive AOE** (Jean's Bomb).
- **`firetrail.gd`** (script-only, instanced from `player.gd`) — a fire patch dropped at the player's feet while moving (**Firewalker** upgrade): plays the fire animation, burns enemies in range on a tick, rotates to the movement direction, renders **under** the actors (`z_index`), and fades after a per-stack duration.
- **`interact_ui.gd`** — one big screen-space **SUMMON** button pinned to the **left edge** (phone-friendly: fixed spot, thumb-sized, doesn't drift with the camera). It appears only when a not-yet-used statue is within range and targets the nearest one; statues themselves just join the `"statue"` group and expose `poate_invoca()` / `invoca()`.
- **`statues.gd`** — spawns statues procedurally, chunk by chunk like trees and rocks: **3% of chunks get exactly one**, at a deterministic position that keeps clear of nearby trees **and rocks** (crowded chunks simply get none, costing ~0.4%). Every run also **starts you at a random point** in the infinite world (`spawner.gd`, saved to `GameSettings.run_spawn`), so the map is different each time without touching the generation math.
- **`statue.tscn`** (`StaticBody2D` + `statue.gd`) — an interactive world statue with a **visually-editable** collision box. Its look is rolled at spawn from three variants (`harta/Statue Version 1..3.png`) with weights **60% / 30% / 10%** — edit the `VARIANTE` list at the top of `statue.gd` to change the odds. When the player is near, a small **"Summon"** button appears. Pressing it plays a one-shot sequence: alert symbol → statue **sinks into the ground** (its collision disables) → **screen shake** (camera-offset earthquake) → a **boss rises slowly from the ground**. All timings/offsets are `@export`.
- **`garda.tscn`** (`CharacterBody2D` + `garda.gd`) — the **"Garda" boss**, spawned *only* by the statue's Summon. Slower and much tankier than a normal enemy, walks toward the player using **8-directional** animations, and **throws lightning balls** from range on a cooldown. Static `garda_0.png` shows while it's rising; the walk animation kicks in once it moves. **(2026-07-19) Special attack:** every **10s** it fires the normal attack **3 times in a fast burst** (`special_interval` / `special_shots` / `special_gap`, ~0.12s apart). The burst is driven by a counter in `_physics_process`, **not `await`** — a coroutine would resume on a freed node if the boss dies mid-burst. It re-reads the player each shot, so the burst tracks you. **(2026-07-22)** What it throws is no longer a lightning ball but a **spinning police baton** — same script and scene, new frames (see below).
- **`lightning.tscn`** (`Area2D` + `lightning.gd`) — the boss's ranged projectile (still called *lightning* in code): since 2026-07-22 a **spinning police baton** with a **circle hitbox**, flies toward the player, and only damages group `"player"` (`take_damage`). **16 frames × 22.5° = one full turn**, played at 24 fps (a turn every 0.67s), each frame carrying a **2px violet outline**. The sprite is scaled to **0.6** so the baton reads about the size of the old ball and roughly matches the 30px hitbox radius.
- **Boss art** lives in `boss/` (walk GIFs split into `walk_<dir>_<i>.png` frames + the lightning-burst frames); the alert symbol in `Upgrades/symbol_alert_002_large_red/`. New GIFs are split to PNG with PowerShell + `System.Drawing`; **open the project in Godot once to import new PNGs** before they render (art is loaded at runtime with `load()`).

## Weapons
Picked in the main menu (`GameSettings.weapon_type`, read by `player.gd` on `_ready`). All four share the **same base stats** (`bullet_damage` 10, `fire_interval` 0.5, `bullet_speed` 700) — `weapon_type` only changes *behaviour*, in four places in `player.gd`:

| Weapon | How it fires |
|---|---|
| **Pistol** | One bullet at the nearest enemy. Nothing else. |
| **Mage Staff** | The same bullet, re-skinned as an animated orb, that **explodes on impact** (radius 110, damage = 60% of `bullet_damage`). |
| **Extinguisher** | No bullets at all — an **aura** pulses around you every `fire_interval`, hitting *every* enemy within `aura_base_radius + level × aura_growth`, for `aura_damage + 50% of bullet_damage` (15 at start). |
| **Cursed Sword** | No bullets — a **melee slash** in the direction you're **facing** (`_facing`), hitting *every* enemy inside a fixed rectangle that runs from the player out to the animation's furthest pixel, for `sword_base_damage + bullet_damage`. Built on the **Firewalker model**: `sword_size` is the width in px, and the rectangle is *measured from the art* at startup, so art and hitbox cannot drift apart. Starts **slow** (`fire_interval × sword_slow_start` when picked) and speeds up with attack-speed upgrades. The slash animation is a **child of the player**, so it follows you (feels like the sword is always in hand), and draws **under** him. Scales with damage / crit / knockback / instakill / weapon-size upgrades. |

**Collision:** everything is on the default layer/mask (layer 1). Bullets (Area2D) detect enemies (CharacterBody2D) via `body_entered` and filter with `is_in_group("enemy")`, so no manual collision-layer setup is needed yet.

Shots crossing the head or legs connect with nothing. Homing hides it by aiming at the centre; making the shape a body-sized capsule would fix it properly, but it also changes how tightly enemies pack and how close they get to you — Răzvan's call.

## Current state (2026-07-21, screen shake fix)
- 🐞 **The screen shook non-stop at high fire rates.** Every crit adds `0.35` trauma and trauma decays at `shake_decay = 4.0`/s, so past **~11.4 attacks per second** it accumulated faster than it drained, pinned at 1.0 and never stopped. Răzvan's recording showed **12.92 attacks/s with 9 projectiles at 43% crit** — with that many projectiles essentially every volley crits, so 12.92 × 0.35 = 4.52 > 4.0. Fixed with a **minimum gap of 0.12s** between shake impulses (`SHAKE_MIN_GAP`), capping the inflow at ~2.9/s. **Nothing changes below ~8 attacks/s** — an isolated crit feels exactly as before. Measured with a repro of those stats: trauma 0.94 / 18.3px offset before, steady 0.35 / ~2px after.

## Current state (2026-07-21, difficulty curve)
*Răzvan's complaint: "enemies are too weak after the first 1:30". They were — phase 1 grew **linearly** (`1 + 0.55 × minutes`) while a build grows multiplicatively, so at minute 10 enemies had only **6.5×** health. And their damage never scaled at all: a minute-10 enemy hit exactly as hard as a second-0 one.*
- ✅ **Phase 1 compounds after 1:30.** The first `RAMP_START` = 90s is untouched; past it, health is `pow(HP_GROWTH_PER_MIN, ramp_minutes)` with **1.40 per minute**. Measured on real spawned enemies: **54 HP at 1:30 → 177 at 5:00 → 956 at 10:00** (was 30 → 54 → ~120 → 195).
- ✅ **Enemies now hit harder over time** — new `Difficulty.enemy_damage_mult()`, reaching **×2 at minute 10** (`DMG_GROWTH_PER_MIN = 2^(1/8.5)`) and doubling every 2 minutes in Final Swarm. Applied to contact damage (including Mike's Hedgehog reflect) and to the Guard's lightning orbs. Base 5 damage taken becomes 7 at 5:00 and 10 at 10:00.
- ⚠️ **XP got the same compounding factor as health.** Left linear, minute-10 enemies would have 32× health and drop 4× XP — the upgrade pace would collapse and the run would read as *impossible* rather than *hard*. The xp/hp ratio now stays flat at **1.59** past 1:30, i.e. exactly the balance the game had before.
- 🔧 **Known side effect:** flat-damage effects fall behind — `fire_trail_damage` (5), `frost_trail_damage` (2) and **Panic Button** (100 flat, versus 956 HP enemies at minute 10). Same problem Jean's Bomb just had; they should probably move to a percentage of player damage too.

- ✅ **Forest-only, never desert or the blend.** Every tile of a candidate path is checked against `BiomeMap.desertness_at_chunk()`; the path is drawn only if **all** tiles are pure grass (`== 0`), so it never touches the desert **or** the soft gradient where desert meets forest. Verified at a biome boundary — paths stop with a clean buffer before the sand.
- ✅ **Shape.** 3 tiles across (a solid dirt strip). Runs **vertical or horizontal** (random). Length **random 4–20 tiles**. Roughly **1 path per 10 chunks** (`spawn_chance = 0.1`). Tiles are 64px = one grass cell, on the world grid; rendered at `z = -5` (above grass, below tree shadows/trunks).
- ✅ **Blend is done in-engine from a single tile.** Uses **only `pathblock normal`** everywhere; a small shader (`path_blend.gdshader`) fades each tile's alpha to transparent on the sides that face grass — decided per-tile from whether a path-tile neighbour exists. So the whole path softly melts into the grass on **all four sides and the two ends** (no hard cut, no protruding caps), and corners come out **rounded** for free (`min` of the two edge falloffs). The four pre-made `pathblock x grassblock *` tiles are no longer used. Fade width is `edge_fade` (0..0.5 of a tile).
- ⚙️ **Why a shader and not instance params.** The edge mask is passed via each sprite's `self_modulate` (R=left, G=top, B=right, A=bottom), *not* `set_instance_shader_parameter` — the latter hits Godot's "too many instances using shader instance variables" cap when lots of path tiles are on screen. `self_modulate` (vertex colour) has no such limit. Knobs on the `Paths` node: `spawn_chance`, `min_len`/`max_len`, `tile_px`, `edge_fade`, `load_radius`.

## Current state (2026-07-21, performance pass)
*The game used to crawl ~2 minutes into Final Swarm — 144 FPS → 4 FPS. Measured, not guessed: an instrumented run that jumps straight to 10:00 and logs FPS + a per-script node census every 5s.*
- 🐞 **XP gems were never removed** — the whole cause of the late-game slowdown. In Final Swarm ~30 enemies die per second and you cannot possibly collect them all, so the gems piled up forever: **5390 gems / 17k nodes at 13:00**, each an `Area2D` animating itself every physics frame. Nodes are now flat at ~1500 and the run holds **144 FPS**, via bubbles:
- ✅ **XP bubbles (`xp3.tscn`).** When **`CLUSTER_NR` = 20** gems of the same kind end up within **`CLUSTER_RAZA` = 260px** of each other, they all **fly into their centre and merge into one bubble** worth exactly their total. Nothing is deleted behind your back and no XP is lost — you *see* it happen. Bubbles merge with each other by the same rule, so however long the run goes, the gem count stays low on its own (measured: ~80 gems + ~70 bubbles, steady, with single bubbles worth 26 000 XP). The **rare xp2 gem never merges** (Răzvan's call — it should stay recognisable). ⚠️ The check runs on the **newly dropped** gem via `call_deferred`, **not** in `_ready`: whoever spawns a gem sets its position *after* `add_child`, so in `_ready` it is still at (0,0) and finds no neighbours — that bug made merging silently never happen.
- ✅ **The bubble is xp2's art, repainted red at runtime** (`_textura_rosie` in `xp.gd`): every pixel gets hue `NUANTA_BULA` (0.0 = red) and keeps its own saturation and value, so the spiral keeps its shading — same drawing, different colour, processed once per run and cached in a `static`. There is deliberately **no `xp3.png` file**: swap `xp/xp2.png` and the bubble follows automatically, with no reimport. A red *overlay* was tried first (additive and mixed) and rejected — the orb stayed blue, or went muddy. See `debugging/geme_xp1_xp2_xp3.png` for how the three read on grass.
- 🐞 **Thunder God spawned one arc per enemy in range, per impact.** In a crowd that is quadratic: **4400 live arcs**, each an `AnimatedSprite2D` with a per-frame `_process`, dragging 300 enemies down to **6 FPS**. Two caps in `player.gd`, both **visual only — every enemy in range still takes the damage**: `THUNDER_MAX_ARCE = 10` arcs drawn per discharge, and `THUNDER_MAX_ARCE_VII = 60` arcs alive on screen in total. Past ~10 overlapping arcs you can't read them anyway.
- ✅ **Enemy hit-flashes are a float countdown, not a Tween** (`enemy.gd`). One `Tween` object per hit is clean until an aura + Thunder God hit 300 enemies several times a second — thousands of allocations per second for a 0.12s colour fade. `_flash_time`/`_flash_dur`/`_flash_color` are drained in `_physics_process` and lerp to the current tint; behaviour is identical (and now the tint tracks the freeze colour *live*). Also: enemies **cache the player node** instead of doing a group lookup every frame (×300), and only call `anim.play()` when the direction actually changed — the same rule `player.gd` already followed.
- ✅ **`Fx` has budgets** (`fx.gd`): at most **45** damage numbers, **25** spark bursts and **35** glow flashes alive at once; requests past the cap are skipped. An aura pulse catching 200 enemies asked for 200 `Label`s in one frame. ⚠️ The counters are reset in `_world()` whenever the current scene changes — the tweens that decrement them belong to nodes in the old scene and die with it, so without the reset the budgets would stay stuck full and **all effects would vanish after a few restarts**.
- ✅ **`Audio.play()` rate-limits per sound name** (`MIN_GAP_MS = 45`). The hit/death sounds are currently missing from `SFX` so nothing changes today, but when they are added back, Final Swarm would ask for them hundreds of times a second — a wall of noise and a pointless cost.
- ℹ️ **Measured result** (worst case: 300 enemies, the spawner's cap, plus Thunder God): **6 FPS → 90-144 FPS**. With a fast-killing build, a full run stays pinned at the refresh rate. The 300-enemy cap itself (`spawner.max_enemies`) is **not** the problem — 300 bodies with no Thunder God ran at 118-144 FPS even before this pass.

### Y-sorting rule (read before adding any world object)
The player's sprite is **centred** on its sort point, so it extends ~64px **below** it. An object whose art ends exactly at its own sort line can never fully cover the player — the legs stay sticking out underneath. Trees solve this with `sort_anchor` (art hangs 73.8px below the sort line) and the statue with `ACOPERIRE_JOS = 74`. **Any new object that should occlude the player needs its art to hang more than ~64px below its sort line.** If you move that art, move its `CollisionShape2D` by the same amount or the hitbox detaches from the sprite.
